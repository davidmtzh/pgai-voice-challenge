# Architecture

This project uses Twilio Voice to place outbound calls to the Pretty Good AI assessment line, and Twilio Media Streams to send live call audio to a Python FastAPI WebSocket server. Because the FastAPI app runs locally during development, ngrok provides a temporary public HTTPS/WSS URL so Twilio can reach the `/voice` and `/media-stream` endpoints. The FastAPI server acts as the live audio bridge between Twilio and the OpenAI Realtime API: it receives phone-call audio from Twilio, forwards it to OpenAI Realtime, receives the generated patient voice response, converts the audio into Twilio-compatible 8kHz G.711 μ-law format, and streams it back into the call.

I chose this architecture because the challenge prioritizes coherent voice conversations, realistic turn-taking, and evidence from real calls over production-grade infrastructure. OpenAI Realtime avoids stitching together separate speech-to-text, LLM, and text-to-speech services, which reduces latency and helps the patient bot respond more naturally. Twilio provides reliable telephony, call recording, and call control, while the scenario system makes it easy to run repeatable patient tests such as scheduling, insurance questions, noisy callers, urgent symptoms, medication refill edge cases, and third-party caregiver calls. In a production version, I would deploy the FastAPI server to a stable cloud backend such as Railway or Azure instead of using ngrok.
